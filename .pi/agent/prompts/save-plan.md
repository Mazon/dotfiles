You are executing the `/save-plan` command. Your job is to extract the plan generated in the previous messages, save it to a file, and populate the task tracker based on the tasks defined in the plan's waves.

If the user provided an argument (`$ARGUMENTS`), use that as the filename (ensure it starts with `.pi/plans/`). If not, generate a descriptive slug.

**Step 1 — Extract and Save the Plan**
1. Look back in the conversation history for the most recent markdown plan generated by the Plan agent.
2. Determine a descriptive slug for the plan filename (e.g., `.pi/plans/plan-<slug>.md` or `.pi/plans/task-<id>-<slug>.md`).
3. Use the `write` tool to save the full plan content (everything before the `## Planning Summary` section) to that file path. Ensure the file contains the Mermaid diagrams, Detailed Specifications, etc.

**Step 2 — Create the Tasks**
Use the `TaskCreate` tool to create a task in the task tracker for EACH task listed in the plan (e.g., the items with `- [ ]` in the wave sections).
For each task:
- `subject`: The title of the task from the plan.
- `description`: The detailed specification for the task extracted from the plan, PLUS a reference to the plan file (e.g., *"See full architectural context and diagrams in .pi/plans/plan-<slug>.md"*).
- `agentType`: "Do"
- `metadata`: `{ "planPath": ".pi/plans/plan-<slug>.md" }`

**Step 3 — Establish Dependencies**
Review the dependency graph in the plan. Use the `TaskUpdate` tool to set the `addBlockedBy` relationships on the newly created tasks so that tasks in Wave 2 wait for Wave 1, etc., accurately reflecting the DAG.

**Step 4 — Present Summary**
List the tasks that were created along with their task IDs, confirm the plan file was saved to the disk, and suggest the user can start execution using `/do <task-id>` or `/tasks` to view the board.
